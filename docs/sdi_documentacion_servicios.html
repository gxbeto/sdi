Orden conceptual de las operaciones

1. Para comprar un producto, primero hay que crearlo (la creación inicializa su stock en cero).
2. Para vender un producto, primero hay que reservarlo (la venta se concreta confirmando la reserva; si se cancela, se libera).

documento_ref vs comprobante

CampoOrigenFormatoEjemplosUso
documento_ref Documento externo recibido desde Ventas o Compras ^[A-Z]{2,3}-\d{3}-\d{6}$ FV-001-000123 · OC-001-000555 Trazabilidad externa
comprobante Comprobante interno generado por Stock en cada operación ^STK-\d{3}-\d{6}-\d$ STK-001-000001-6 Trazabilidad interna. Prueba de que la operación anterior se realizó
El último dígito del comprobante es un dígito verificador (algoritmo de Luhn sobre los 9 dígitos de serie + correlativo). Permite validar de entrada si un comprobante es oficial sin consultar el sistema. El comprobante no reemplaza a documento_ref: conviven en las respuestas y en los movimientos.

Política numérica (x-numeric-policy)

GrupoTipo JSONPrecisiónRegla
CantidadesnumberHasta 3 decimales (multipleOf: 0.001)Entradas > 0 · saldos y salidas ≥ 0
PreciosnumberHasta 2 decimales (multipleOf: 0.01)≥ 0 · sin campo moneda
ImpuestonumberHasta 2 decimalesEntre 0 y 100

Los valores numéricos viajan como números JSON ("cantidad": 2.5), nunca como string ni como entero forzado.

Formato de errores

{ "estado": "ERROR", "codigo": "CODIGO_TECNICO", "mensaje": "Descripción clara del error." }

Todas las respuestas de error usan el schema ErrorResponse. Ver el catálogo de errores.

Más combinaciones válidas de include

Compras · Procesar evento de compra

Idempotencia

Si el mismo mensaje_id ya fue procesado, la respuesta es estado_procesamiento: "DUPLICADO" (devolviendo el comprobante ya emitido) y no se vuelve a incrementar el stock.

Reservas · Crear reserva de stock

POST/stock/reservas operationId: crearReserva IDEMPOTENTE · clave: documento_ref
crear_reserva(payload: ReservaCreate) -> ReservaCreateOut  ·  servicio: inventory.crear_reservas

Reserva stock para uno o más productos. Es el paso previo obligatorio a la venta: incrementa cantidad_reservada (sin descontar el total), crea reservas en estado PENDIENTE y registra movimientos RESERVA. Genera un comprobante interno por operación, compartido por todos los ítems.

Ventas · Confirmar venta completa

POST/ventas/confirmar operationId: confirmarVenta IDEMPOTENTE · clave: documento_ref
confirmar_venta(payload: VentaConfirmarIn) -> VentaConfirmarOut  ·  servicio: inventory.confirmar_venta

Confirma una venta con múltiples ítems usando las reservas PENDIENTES del mismo documento_ref. Cada ítem debe coincidir en producto y cantidad con su reserva.
